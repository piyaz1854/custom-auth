About

{{ title }} is a simple but solid backend project that shows how to build your own authentication and authorization system without relying on Django’s built-in tools.

The main idea is pretty clear: users get roles, roles define what they can or can’t do with different parts of the system (we call them “business elements”). All permissions are stored in the database and checked every time someone makes a request.

Passwords are safely hashed using bcrypt before saving — so no raw passwords ever hit the database. For login, the system uses JWT (JSON Web Tokens), which means no sessions or cookies, just clean token-based authentication.

This project is great if you want to understand how things work under the hood — from managing user accounts and tokens, to controlling access through roles and custom middleware.

Tech Stack: Python, Django, Django REST Framework, PostgreSQL, pyjwt for working with tokens, and bcrypt for hashing passwords.

How it works

Everything starts with the user. They can create an account by entering their name, email, and password. After that, they can log in, edit their profile, or deactivate their account if they no longer want to use it. Instead of deleting users from the system, we simply mark them as inactive.

After logging in, the server gives the user a token. This token must be sent with every request to access protected parts of the system. There are no sessions or cookies involved — the whole system is based on tokens.

Each request goes through a custom piece of code called middleware. It checks if the token is valid, finds the right user, and connects that user to the request. So when the system handles the request, it already knows who is asking.

Then comes access control. Every user has a role — like overlord, commander, or just a scout and novice. Each role has its own set of permissions that define what actions are allowed. For example, can the overlord see something, change it, or delete it? And if they can delete — only their own data or everyone’s?

These rules are stored in the database and checked every time someone tries to do something important. If the user doesn’t send a token, they get a 401 error (not authorized). If they send a token but don’t have permission, it’s a 403 (forbidden). If everything is okay, the system allows the action.

In the end, the result is a simple and flexible system where you decide who gets access to what — and the rules are clear, easy to manage, and live in the database, not buried in the code.

Features

  • No default Django login or register pages — everything is built manually from the ground up (somewhere used built-in :0).
  • Authentication is fully JWT based.
  • Roles and permissions are stored in the database.
  • Custom middleware checks each request and makes sure the user is allowed to access what they’re asking for.
  • When a user deletes their account, it's just turned off (soft delete).
